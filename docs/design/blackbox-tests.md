
# Black-box functional tests

> **When to read**: adding a new black-box scenario, modifying the test runner, troubleshooting a failing scenario, or adding a new feature that needs functional tests. **Not needed for**: writing Go unit tests or integration tests (use `testing.md`), understanding protection behavior (use `protections.md`).

Black-box tests verify barbacana's HTTP-observable behavior by driving real requests through a compiled binary. The test suite uses [Hurl](https://hurl.dev) for declarative HTTP assertions and a Go test runner (`runner_test.go`) that boots the WAF once per scenario.

## Design principles

1. **Scenario-based structure.** A test case is a (config, request, expectation) triple. Each scenario has its own config file and its own Hurl tests. Scenarios are isolated — adding a new one never modifies an existing one.

2. **HTTP-observable behavior only.** Tests assert on status codes, response headers, and response bodies. Internal behavior — audit logs, Prometheus metrics, temp file cleanup, goroutine cancellation — belongs in Go unit and integration tests.

3. **No external dependencies.** No containers, no Docker Compose, no external images, no network setup. The suite needs only the compiled `barbacana` binary, Go (for the mock upstream), and `hurl`.

4. **One WAF boot per scenario.** The runner starts the WAF with the scenario's config, runs all Hurl files for that scenario, then tears down. A single mock upstream runs for the entire test run.

5. **Port choices.** WAF listens on `:18080`, upstream on `:19000`. These are chosen to avoid conflicts with common development tools (8080, 3000, 5432, etc.).

6. **Go-managed process lifecycle.** The runner uses `context.WithCancel` to manage child processes and `net.DialTimeout` polling to wait for port readiness. No PIDs, no sleep, no shell traps.


## How to add a new scenario

1. Create a new directory under `tests/blackbox/scenarios/` — the name should describe the behavior being tested.
2. Write `config.yaml` using `version: v1alpha1`, `listen: ":18080"`, `upstream: http://localhost:19000`. Include only fields relevant to the scenario; rely on defaults for everything else.
3. Write one or more Hurl files in `tests/`. Number them (`01-`, `02-`) for predictable execution order. Each file covers one concern.
4. If the scenario needs generated fixtures (binary files, compressed payloads), add a `setup.sh` that creates a `tests/fixtures/` directory. The runner executes it before running Hurl.
5. If the scenario uses OpenAPI validation, add `spec.yaml` in the scenario directory and reference it in `config.yaml` relative to the repo root.
6. Run `make build && make test-blackbox` to verify.

No changes to `runner_test.go`, the Makefile, or any other scenario are required.

## Hurl conventions

- Use `{{host}}` for the WAF base URL (passed as `--variable host=http://localhost:18080`).
- Use `{{upstream}}` if you need to reference the upstream directly (passed as `--variable upstream=http://localhost:19000`).
- Never hardcode `localhost:18080` or `localhost:19000` in Hurl files.
- Number files for ordering: `01-clean-get.hurl`, `02-sqli-blocked.hurl`.
- One concern per file.
- Always assert on the HTTP status code.
- For blocked requests (403), assert on the JSON error body:
  ```hurl
  HTTP 403
  [Asserts]
  jsonpath "$.error" == "blocked"
  jsonpath "$.request_id" isString
  jsonpath "$.request_id" != ""
  ```
- For pass-through requests (200), assert on the upstream echo:
  ```hurl
  HTTP 200
  [Asserts]
  jsonpath "$.method" == "GET"
  jsonpath "$.path" == "/expected/path"
  ```

## Mock upstream

The mock upstream (`tests/blackbox/upstream/main.go`) is a Go HTTP server on `:19000`. Every request returns `200 OK` with a JSON body echoing the request:

```json
{
  "method": "POST",
  "path": "/users",
  "query": "id=42",
  "headers": {"Content-Type": "application/json", "...": "..."},
  "body_size": 128
}
```

The upstream also sets `Server: MockUpstream/1.0` and `X-Powered-By: Go` response headers so that header-stripping scenarios can verify the WAF removes them.

Headers are returned as `map[string]string` (first value per key), enabling assertions like `jsonpath "$.headers.X-Custom"`.

## Coverage strategy

Every protection implemented must have black-box coverage across three dimensions: **attack blocked**, **disable pass-through**, and **detect-only pass-through**. This section defines the patterns and scoping rules.

### Test dimensions

Each testable protection needs scenarios from three angles:

1. **Attack blocked** — the protection is enabled (default), and a malicious or invalid request gets the expected rejection status (403, 405, 413, 415, 422, 431, etc.).
2. **Disable pass-through** — the protection's canonical name appears in the route's `disable` list, and the same attack payload passes through to the upstream (200).
3. **Detect-only pass-through** — the route has `mode: detect_only`, and the attack payload is proxied to the upstream (200) instead of blocked.

Clean-traffic scenarios (valid requests pass with the protection enabled) overlap with the "attack blocked" scenario configs — every attack scenario should include at least one clean request to prove normal traffic is not affected.

### Scoping rules

- **CRS categories**: test one representative payload per *category* (e.g. one sqli, one xss, one rce, one lfi), not per sub-protection. Sub-protection coverage belongs in Go unit tests backed by the CRS test suite. The Hurl test proves the category wiring works end-to-end.
- **Native protections** (protocol hardening, normalization, request validation, resource limits, file uploads, OpenAPI, CORS): test each individual protection since they have distinct blocking logic.
- **Security headers** (injection and stripping): test each header individually — the config allows disabling per-header, so each needs a scenario proving the default and a scenario proving the disable.
- **Slow-request and HTTP/2 limits** (B10): excluded from Hurl. These are Caddy server-level timeouts, not triggerable from declarative HTTP assertions. Covered in Go integration tests.
- **Evaluation timeout**: excluded from Hurl. Requires payloads that take >N seconds to evaluate, which is fragile in CI. Covered in Go integration tests.

### Disable test grouping

Disable scenarios do not need one scenario per protection. Instead, test one representative per *type* of disable mechanism:

- Disable a CRS category (all sub-protections pass)
- Disable a single CRS sub-protection (only that one passes, others still block)
- Disable a native protocol protection
- Disable a normalization protection
- Disable a request validation protection
- Disable a header injection protection
- Disable a header stripping protection
- Disable an OpenAPI check
- Disable a file upload check

Each disable scenario uses 1-2 Hurl requests: one proving the attack passes, one proving a *different* protection in the same category still blocks.

### Detect-only test grouping

Detect-only scenarios need one representative per protection group to prove the mode works across the pipeline:

- CRS-backed attack (e.g. sqli)
- Native protocol attack (e.g. crlf)
- Request validation rejection (e.g. oversized body)
- OpenAPI rejection
- File upload rejection
- Normalization attack (e.g. double-encoding)

Detect-only mode does not apply to security headers (injection/stripping) — those are response modifications, not request blocking decisions.

### What NOT to test in Hurl

- Individual CRS sub-protections (e.g. all 22 sqli variants). The CRS test suite payloads cover those in Go unit tests. Hurl tests one payload per category.
- Audit log fields, metric labels, temp file cleanup, goroutine lifecycle — these are internal behaviors tested in Go.
- Timing-dependent behaviors (slow-request timeouts, evaluation timeouts).
- Config validation error messages.

## What belongs in black-box tests vs Go tests

| Behavior | Black-box (Hurl) | Go unit/integration |
|---|---|---|
| Attack payload blocked (status 403) | Yes | Yes |
| Attack payload passes in `detect_only` mode (status 200) | Yes | Yes |
| Security headers present in response | Yes | — |
| Security headers stripped from response | Yes | — |
| CORS preflight headers correct | Yes | — |
| Path rewrite visible at upstream | Yes | — |
| OpenAPI validation returns correct status | Yes | Yes |
| Content-type gating returns 415 | Yes | Yes |
| Audit log JSON format and fields | — | Yes |
| Prometheus metric names and labels | — | Yes |
| Temp file cleanup after request | — | Yes |
| Goroutine cancellation on timeout | — | Yes |
| CRS evaluation timeout behavior | — | Yes |
| Config validation error messages | — | Yes |
| Body spooling to disk | — | Yes |
| SIGHUP reload applies new config | — | Yes |

## Running locally

```bash
# Build the binary, then run black-box tests
make build && make test-blackbox

# Equivalently:
make build
go test -tags=blackbox ./tests/blackbox/ -v -count=1

# Run a single scenario manually
./barbacana serve --config tests/blackbox/scenarios/blocking-sqli/config.yaml &
hurl --test \
  --variable host=http://localhost:18080 \
  --variable upstream=http://localhost:19000 \
  tests/blackbox/scenarios/blocking-sqli/tests/*.hurl
kill %1
```

## CI integration

Black-box tests run after unit and integration tests in `.github/workflows/ci.yml`:

```yaml
- name: Unit tests
  run: go test -race ./...

- name: Integration tests
  run: go test -tags=integration -race ./internal/pipeline/...

- name: Black-box tests
  run: make test-blackbox
```

The CI job installs Hurl using the official install script from `https://hurl.dev`.
